# 메뉴 구성 및 책임 분리

상태: Active  
작성일: 2026-03-18  
목적: 메뉴별 구현 방식을 비교 가능한 형태로 정리하고, 어떤 메뉴가 어떤 수준으로 분리되어 있는지 설명

---

## 1. 목적

황총무는 메뉴가 많기 때문에 "한 메뉴에서 익힌 패턴이 다른 메뉴에도 그대로 적용되는가?"가 중요하다.  
이 문서는 메뉴별 책임 분리 수준을 비교해 구조적 일관성과 편차를 함께 기록한다.

---

## 2. 공통 메뉴 패턴

대부분 메뉴는 아래 패턴 중 하나를 따른다.

### 패턴 A. 생성 페이지 + 상세 페이지

예:

- meeting
- calc
- habit
- daily
- diet

의미:

- 진입 비용을 낮추고
- 생성 후 상세 화면에서 본작업 수행

### 패턴 B. 허브 + 워크스페이스/상세

예:

- account-book
- schedule

의미:

- 목록/허브 화면에서 객체 선택
- 상세 화면에서 실제 작업

### 패턴 C. 포털 + 하위 모드 분기

예:

- game

의미:

- 하나의 메뉴 안에 여러 사용 모드를 품고 있음

---

## 3. 메뉴별 책임 분리 비교

| 메뉴 | 생성/목록 | 상세 컨테이너 | 로직 훅 | 데이터 레이어 | 분리 수준 |
| --- | --- | --- | --- | --- | --- |
| meeting | 있음 | 있음 | 강함 | Supabase 직접 + 훅 | 높음 |
| calc | 있음 | 있음 | 강함 | Supabase + RPC | 높음 |
| account-book | 허브형 | 있음 | 중간 | localStorage storage | 높음 |
| habit | 있음 | 있음 | 강함 | Supabase 직접 + 훅 | 중상 |
| daily | 있음 | 있음 | 낮음 | localStorage storage | 중상 |
| diet | 있음 | 있음 | 낮음 | Supabase 직접 | 중간 |
| game | 포털형 | 있음 | 강함 | Supabase realtime | 중상 |
| schedule | 목록형 | 있음 | 중간 | services/schedule | 높음 |
| portfolio | 문서형 | 있음 | 낮음 | 정적 데이터 | 중상 |

## 4. 각 메뉴의 구조적 특징

### meeting

- 페이지와 훅 분리 좋음
- 상세 UI가 `detail/`로 잘 쪼개짐
- 비즈니스 로직이 `useRoom`에 집중

### calc

- 저장, 계산, UI 조립이 각각 분리
- 계산 엔진(`useCalculator`)이 독립적이라 가장 해석이 쉬운 편

### account-book

- 허브 / 상세 / 설정 / 잠금 레이어로 나뉨
- 모델은 복잡하지만 저장 레이어가 분리되어 있음
- 복잡도는 가장 높음

### habit

- `MonthlyTracker`와 `useMonthlyTracker`로 컨테이너가 잘 나뉨
- 댓글/랭킹/체크리스트가 한 화면에 모여 있어 기능 응집도가 높음

### daily

- 훅 분리는 약하지만 storage 추상화가 좋아서 유지보수성이 확보됨
- 페이지 자체 로직이 큰 편

### diet

- `DietMainContent`에 많은 로직이 모여 있음
- 차트/분석/입력 컴포넌트는 분리됐지만 컨테이너가 두꺼움

### game

- 로비와 방 구조가 다름
- `useGameRoom`이 강력하지만 게임 유형이 늘어나면 더 추상화가 필요

### schedule

- services 레이어가 명확함
- 목록/생성/상세/칸반 역할이 잘 분리
- 운영 도구치고 구조가 안정적

### portfolio

- 문서형/콘텐츠형 구조
- 정적 데이터 중심
- 서비스 로직보다는 정보 아카이브 성격

---

## 5. 현재 구조에서의 좋은 패턴

재사용 가치가 큰 패턴:

- `useXxx` 훅에 흐름 로직 집중
- `storage.ts`로 브라우저 저장소 추상화
- `services/*.ts`로 외부 데이터 접근 분리
- 상세 페이지에서 UI 조각을 기능 단위로 분해

이 패턴은 이후 신규 메뉴를 추가할 때 기준으로 삼을 수 있다.

---

## 6. 현재 구조에서의 편차

### 편차 1. 데이터 접근 방식

- 어떤 메뉴는 service 레이어 사용
- 어떤 메뉴는 페이지/훅에서 supabase 직접 호출
- 어떤 메뉴는 localStorage 직접 관리

### 편차 2. 컨테이너 두께

- calc, schedule는 비교적 얇음
- diet, daily는 페이지/컨테이너가 두꺼움

### 편차 3. 상태 소유 위치

- 훅에 집중된 메뉴와
- 컴포넌트 내부에 집중된 메뉴가 공존

---

## 7. 문서상 권장 기준

새 메뉴 또는 리팩토링 시 권장 기준:

1. 생성/목록 페이지와 상세 작업 페이지 분리
2. 비즈니스 흐름은 훅 또는 service에 분리
3. 브라우저 저장소는 반드시 `storage.ts` 추상화
4. 외부 데이터는 service 또는 repository 계층으로 수렴
5. 상세 페이지는 컨테이너 + 프레젠테이션 구조로 구성

---

## 8. 결론

황총무는 이미 여러 좋은 분리 패턴을 갖고 있지만, 메뉴마다 도입 시점이 달라 구조 편차도 분명하다.  
실무적으로는 이 편차를 "문제"로만 볼 게 아니라, **어떤 패턴이 유지보수에 강했는지 식별하고 표준화하는 기회**로 보는 것이 좋다.
