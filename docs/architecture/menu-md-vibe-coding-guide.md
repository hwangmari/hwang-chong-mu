# 메뉴별 MD 아키텍처 작성 기준

상태: Draft  
작성일: 2026-04-01  
목적: 메뉴별 상세 문서를 작성할 때, AI 기반 바이브 코딩 전환 관점에서 어떤 구조와 기준으로 설명할지 통일

---

## 1. 한 줄 핵심 요약

핵심 메시지:

> AI 기반 바이브 코딩 전환을 위해 MSA + 컴퓨팅 리소스 아키텍처 중심으로 개발 체계를 재구성하고, 비즈니스 이해 기반으로 운영하는 것이 핵심

메뉴 문서는 기능 설명만 적는 문서가 아니다.  
각 메뉴가 이 전환 흐름 안에서 어떤 구조적 역할을 하는지 설명해야 한다.

---

## 2. 전체 구조

### 2-1. 왜 하는가

이 기준 문서는 다음 목적을 전제로 한다.

- AI 기반 개발 전환
- 개발 생산성 극대화
- 코드 품질과 운영 안정성 확보
- 팀 역량 증명 및 이후 지속 가능성 판단

핵심 전제:

- 코드의 80~90%는 AI가 생성할 수 있다
- 사람은 생성보다 검증, 구조 판단, 운영 안정화에 더 집중해야 한다
- 따라서 메뉴 문서도 "기능 설명"보다 "구조 설명" 비중이 커져야 한다

### 2-2. 무엇이 바뀌는가

#### 기존 방식

- 사람이 직접 구현 중심
- FE / BE 역할 분리
- 기능 단위 개발

#### 바이브 코딩 전환 이후

- AI가 코드 생성, 사람이 검증
- 부분 수정 중심이 아니라 전체 구조 기반 개발
- 비즈니스 중심 명령 기반 개발
- 구조와 아키텍처가 성능과 품질을 좌우

즉, 메뉴 문서는 "이 화면에 뭐가 있다"보다 아래를 설명해야 한다.

- 이 메뉴가 어떤 비즈니스 흐름을 담당하는가
- 어떤 경계를 기준으로 분리되어 있는가
- AI가 작업할 때 무엇을 먼저 이해해야 하는가

---

## 3. 핵심 성공 요소

중요도 순서:

### 3-1. 컴퓨팅 리소스 아키텍처

가장 중요한 요소는 코드 자체보다 리소스 구조다.

- 리소스 구조가 성능과 안정성을 결정
- 클라우드 환경 기반 설계가 필수
- 단순 구현보다 인프라/런타임 구조가 더 중요

메뉴 문서에는 가능하면 아래 관점을 포함한다.

- 이 메뉴가 어떤 저장소와 런타임에 의존하는가
- 병목이 생길 수 있는 지점은 어디인가
- 클라이언트/서버/DB/RPC 중 어디가 핵심 자원인가

### 3-2. MSA 구조

두 번째는 서비스 분리 구조다.

- 서비스 단위 분리
- 변경 영향 최소화
- AI 코드 생성 시 안정성 확보

메뉴 문서에는 아래 질문을 반영한다.

- 이 메뉴는 독립 서비스처럼 볼 수 있는가
- 어떤 모듈/도메인과 강하게 결합돼 있는가
- 변경 시 영향 범위는 어디까지인가

### 3-3. 비즈니스 이해도

세 번째는 비즈니스 흐름 이해다.

- 비즈니스 흐름 기반 설계 필수
- 단순 기능 개발은 위험
- 비즈니스 스코어 기반 개발이 중요

핵심 문장:

> 비즈니스를 모르면 AI도 잘 못 쓴다

즉, 메뉴 문서는 반드시 기능보다 먼저 아래를 설명해야 한다.

- 이 메뉴를 왜 쓰는가
- 실제 사용자 맥락은 무엇인가
- 어떤 판단 기준이 중요한가

### 3-4. 프로젝트/프레임워크 이해

구조 이해 없이 작업하면 AI 생산성도 떨어진다.

- 프로젝트 구조 이해 없이 수정하면 비효율이 커진다
- 필요한 정보만 경량화해서 정리하는 것이 중요하다
- 퍼포먼스/아키텍처 심화 내용은 별도 문서에서 관리한다

따라서 메뉴 문서는 "필요한 만큼만" 구조를 보여줘야 한다.

- 핵심 라우트
- 핵심 파일
- 상태/데이터 흐름
- 저장 방식
- 외부 의존성

---

## 4. 개발 방식: Vibe Coding Rule

### 4-1. 기본 원칙

- 직접 코드 수정 최소화
- AI에게 재요청하는 방식으로 수정
- 전체 코드 싱크 유지 필수

이 원칙은 메뉴 문서에도 반영되어야 한다.

- 로컬 수정 포인트보다 구조적 수정 포인트를 먼저 설명
- "어디 한 줄 바꾸면 된다"보다 "어떤 책임을 바꿔야 한다"를 우선 정리

### 4-2. 작업 방식

기본 흐름:

1. 요구사항 정의
2. AI에게 명확한 명령 전달
3. 결과 검증
4. 필요 시 재요청
5. 반영

중요한 점:

- 요구사항은 비즈니스 기준으로 정의한다
- 명령은 화면이 아니라 구조 기준으로 전달한다
- 검증은 기능 동작뿐 아니라 전체 싱크를 본다

---

## 5. 메뉴별 MD에 꼭 들어가야 하는 관점

각 메뉴 문서는 최소한 아래 질문에 답해야 한다.

1. 이 메뉴의 비즈니스 목적은 무엇인가
2. 이 메뉴가 전체 서비스 구조에서 맡는 역할은 무엇인가
3. 어떤 데이터/리소스/런타임에 의존하는가
4. 어떤 경계 기준으로 모듈 또는 서비스가 분리되는가
5. AI가 이 메뉴를 수정하려면 무엇을 먼저 이해해야 하는가
6. 변경 시 영향 범위는 어디까지인가
7. 검증 포인트는 무엇인가

---

## 6. 메뉴별 MD 권장 템플릿

아래 템플릿을 공통 포맷으로 사용한다.

### 6-1. 한 줄 요약

- 이 메뉴의 핵심 가치를 한 문장으로 정리

### 6-2. 비즈니스 목적

- 왜 이 메뉴가 필요한가
- 누구의 어떤 문제를 해결하는가

### 6-3. 구조적 역할

- 허브 / 생성 / 상세 / 설정 / 운영 도구 중 어떤 성격인가
- 전체 메뉴 흐름 안에서 어느 단계에 위치하는가

### 6-4. 리소스 아키텍처 관점

- 클라이언트 / 서버 / DB / RPC / 로컬 저장 중 핵심이 무엇인가
- 성능과 안정성을 좌우하는 지점은 어디인가

### 6-5. 서비스/모듈 분리 관점

- 이 메뉴가 독립 서비스처럼 동작하는가
- 강결합된 모듈은 무엇인가
- 변경 영향 범위는 어디까지인가

### 6-6. 비즈니스 이해 포인트

- AI가 작업 전 반드시 이해해야 할 비즈니스 규칙
- 예외 케이스와 운영상 중요 포인트

### 6-7. 프로젝트/프레임워크 이해 포인트

- 핵심 라우트
- 핵심 파일
- 핵심 상태/데이터 흐름
- 저장소 또는 외부 연동 방식

### 6-8. Vibe Coding 작업 규칙

- 어떤 요구사항 단위로 AI에 요청할지
- 어떤 검증 기준으로 결과를 볼지
- 어떤 수준의 변경은 국소 수정이 아니라 구조 수정으로 봐야 하는지

### 6-9. 리스크와 다음 작업

- 현재 구조적 리스크
- 다음 리팩토링 우선순위

---

## 7. 문서 작성 시 주의사항

- 기능 나열형 문서로 끝내지 않는다
- 화면 설명보다 구조 설명을 우선한다
- 코드 세부보다 책임 경계와 데이터 흐름을 먼저 적는다
- AI가 읽고 바로 작업 맥락을 잡을 수 있게 쓴다
- 너무 무겁게 쓰지 말고, "필요한 정보만 경량화"해서 유지한다

---

## 8. 결론

메뉴별 MD는 단순 분석 문서가 아니라, 바이브 코딩 시대의 작업 명세서 역할을 해야 한다.  
핵심은 기능 목록이 아니라 아래 4가지를 한 번에 전달하는 것이다.

- 왜 존재하는가
- 어떤 구조 위에서 돌아가는가
- 무엇이 성공 요소인가
- AI가 어떤 기준으로 작업해야 하는가
