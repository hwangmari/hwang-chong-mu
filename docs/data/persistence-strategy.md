## 1. 요약

황총무는 모든 메뉴를 하나의 저장 방식으로 통일하지 않았다.  
현재는 도메인 성격에 따라 **Supabase**와 **localStorage**를 혼용하는 하이브리드 전략을 사용한다.

---

## 2. 저장 전략 분류

### 2-1. Supabase 사용 메뉴

- 약속 잡기
- N빵 계산기
- 습관 관리
- 체중 관리
- 게임방
- 업무 캘린더

선택 기준:

- 여러 사용자가 같은 데이터를 본다
- 링크 공유 / 방 입장 개념이 있다
- 서버 저장이 필요하다
- 일부 메뉴는 실시간성 또는 RPC가 필요하다

### 2-2. localStorage 사용 메뉴

- 가계부
- 일일 기록

선택 기준:

- 개인 중심 사용
- 빠른 구현
- 서버리스에 가까운 UX
- 기기 내 지속성만 있어도 초기 목적을 달성 가능

---

## 3. 메뉴별 저장소 선택 이유

| 메뉴 | 저장소 | 선택 이유 |
| --- | --- | --- |
| meeting | Supabase | 참가자 협업, 링크 공유, 확정 상태 저장 |
| calc | Supabase + RPC | 멤버/지출 공동 저장, 방 개념, 정산 데이터 유지 |
| account-book | localStorage | 개인/가정용 중심, 빠른 UX 실험 |
| habit | Supabase | 목표/로그 구조, 장기 추적, 잠재적 다중 접근 |
| daily | localStorage | 개인 기록장, 속도, 간편성 |
| diet | Supabase | 날짜별 로그의 지속 저장 |
| game | Supabase + realtime | 실시간 참가자 변화와 방 상태 동기화 |
| schedule | Supabase | 내부 운영 데이터와 CRUD 중심 업무 도구 |

## 4. Supabase 전략

### 장점

- 데이터 공유가 자연스럽다
- 링크 기반 진입이 가능하다
- 서버 저장으로 기기 독립성이 있다
- realtime, RPC 같은 고급 기능을 일부 메뉴에서 활용 가능

### 비용

- 테이블 설계와 권한 정책 관리 필요
- 화면 로직이 DB 구조를 많이 의식하게 될 수 있다
- 메뉴마다 직접 쿼리하는 구조는 결합도를 올린다

### 현재 구조의 특징

- `lib/supabase.js`에서 단일 클라이언트 생성
- 일부 메뉴는 훅에서 직접 DB 호출
- 일정 메뉴는 `services/schedule.ts`로 service 레이어를 도입
- calc는 RPC 중심으로 데이터 교체 전략 사용

---

## 5. localStorage 전략

### 장점

- 빠르고 간단하다
- 서버 의존성이 없다
- 개인용 도구 프로토타이핑에 강하다
- 브라우저 내 즉시 반응형 UX에 유리하다

### 비용

- 기기 종속
- 브라우저 삭제/초기화 시 데이터 유실
- 동기화/백업/공유가 어렵다
- 멀티 유저처럼 보이는 모델도 실제론 로컬에 국한됨

### 현재 구조의 특징

- 가계부는 `storage.ts` 기반 자체 store 모델 보유
- 일일 기록은 key 설계와 월별 데이터 분리를 꽤 체계적으로 구현

---

## 6. 현재 전략의 구조적 장점

- 기능별로 가장 빠르게 성과를 낼 수 있었다
- 모든 기능을 같은 수준의 복잡도로 만들지 않아도 됐다
- 서버가 꼭 필요한 기능만 서버로 보냈다

---

## 7. 현재 전략의 구조적 리스크

### 리스크 1. 사용자 기대치 불일치

사용자 입장에서는 모두 같은 앱인데,

- 어떤 메뉴는 계정/공유/동기화가 되고
- 어떤 메뉴는 브라우저 로컬에만 저장된다

이 차이를 UI에서 충분히 설명하지 않으면 혼란이 생길 수 있다.

### 리스크 2. 백업/복구 불균형

- Supabase 메뉴는 서버 저장
- localStorage 메뉴는 백업 부재

가계부처럼 민감한 데이터가 localStorage에만 있는 것은 장기적으로 부담이다.

### 리스크 3. 기술부채 분산

저장소가 둘로 나뉘면 다음도 둘로 나뉜다.

- 테스트 전략
- 오류 처리 전략
- 마이그레이션 전략
- 운영 이슈 대응 방식

---

## 8. 실무적 권장 방향

### 단기

- localStorage 메뉴에 export/import 추가
- localStorage 기반 메뉴에 "이 브라우저에만 저장됩니다" 안내 추가
- Supabase 접근 패턴을 문서화

### 중기

- service layer가 없는 Supabase 메뉴에 데이터 접근 계층 도입
- 가계부/일일 기록 중 서버 동기화가 필요한 메뉴 선정

### 장기

- 저장 전략 통합 재검토
- 사용자 계정 기반 제품화 여부 결정
- localStorage 메뉴의 Supabase 전환 가능성 검토

---

## 9. 결론

현재 하이브리드 전략은 "실험과 속도" 측면에서는 적절했다.  
하지만 문서와 운영 체계가 없으면 시간이 갈수록 유지보수 비용이 커진다.  
따라서 지금 단계에서 필요한 것은 저장 전략을 바꾸는 것보다, **어떤 메뉴가 왜 이 저장소를 쓰는지 공식적으로 기록하는 것**이다.
