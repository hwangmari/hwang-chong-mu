# 기술 진화 히스토리

상태: Working Draft  
작성일: 2026-03-18  
주의: 이 문서는 커밋 로그 전체가 아니라 현재 코드와 마이그레이션 흔적을 기반으로 재구성한 구조적 히스토리다.

---

## 1. 목적

이 문서는 황총무가 어떤 방향으로 진화해왔는지, 현재 구조가 왜 이렇게 생겼는지를 설명한다.  
실무에서 이 문서는 아래 질문에 답하기 위해 필요하다.

- 왜 어떤 메뉴는 service 레이어가 있고 어떤 메뉴는 없는가?
- 왜 어떤 메뉴는 slug를 쓰고 어떤 메뉴는 단순 id를 쓰는가?
- 왜 localStorage와 Supabase가 공존하는가?
- 어떤 리팩토링이 이미 있었고, 어떤 구조가 아직 과도기인가?

---

## 2. 관찰 가능한 진화 축

현재 코드 기준으로 보면 황총무의 구조 변화는 크게 5개 축으로 읽힌다.

1. 단일 생활 도구 -> 멀티 메뉴 실험실
2. 브라우저 로컬 중심 -> 일부 서버 저장 중심 전환
3. 단순 id URL -> 사람이 읽기 쉬운 공유 주소
4. 개별 화면 구현 -> 공통 컴포넌트와 공통 레이어 도입
5. 기능 구현 중심 -> 문서와 운영 관점 추가

---

## 3. 추정 가능한 진화 단계

### Phase 1. 빠른 기능 실험 단계

특징:

- 생활 문제를 빠르게 푸는 메뉴 단위 기능 개발
- 페이지 내부 로직 비중 높음
- localStorage 또는 직접 Supabase 호출 위주
- 구조보다 기능 도달 속도 우선

흔적:

- 메뉴마다 저장 방식이 제각각
- 일부 메뉴의 컨테이너가 두꺼움
- 직접적인 Supabase 쿼리가 페이지/훅 내부에 존재

---

### Phase 2. 공유/방 개념 도입 단계

특징:

- meeting, calc, game 같이 "링크 공유"가 중요한 메뉴 등장
- 다중 사용자 사용 시나리오 고려
- Supabase 저장 구조 확대

흔적:

- `rooms`, `calc_rooms`, `game_rooms`
- 참가자 개념
- 상세 페이지와 생성 페이지 분리

---

### Phase 3. 주소 품질 개선 단계

특징:

- UUID 직접 노출보다 공유 가능한 링크 품질을 개선
- slug + short_code 전략 도입

흔적:

- `supabase/20250206_add_room_short_code_slug.sql`
- `supabase/20250206_add_calc_room_short_code_slug.sql`
- backfill SQL
- meeting, calc 상세 페이지의 UUID -> slug redirect 처리

해석:

- 제품이 단순 실험을 넘어 "실제 공유되는 서비스"로 진입하면서 주소 품질을 개선한 단계로 볼 수 있다.

---

### Phase 4. 메뉴 간 연결 단계

특징:

- 기능이 서로 독립적이기만 하지 않고 연결되기 시작

대표 사례:

- meeting -> calc

흔적:

- `supabase/20250206_add_calc_room_id_to_rooms.sql`
- `room.calc_room_id`
- 약속 확정 후 정산 방 생성/갱신 로직

해석:

- 이 시점부터 황총무는 단순 도구 모음집이 아니라 "도구 간 워크플로 연결"을 시도하기 시작했다.

---

### Phase 5. 구조화와 UX 단순화 단계

특징:

- 기능 자체는 복잡해졌지만 사용자 입력은 더 단순하게 만들려는 방향
- 공통 문서와 공통 UI 구조 정리 시작

대표 사례:

- 가계부 자연어 입력
- `/docs` 문서 페이지 도입
- 메뉴별 문서 분리

해석:

- 내부 구조는 확장되었고, 이제는 학습 비용과 온보딩 비용을 낮추려는 단계로 넘어가고 있다.

## 4. 메뉴별 히스토리 단서

### meeting

- 방 생성/참여 모델 초기부터 존재한 것으로 보임
- 이후 slug + short_code가 도입됨
- 나중에 calc 연동이 붙으면서 단순 일정 조율 도구를 넘어 워크플로 도구가 됨

### calc

- 초기에는 방/멤버/지출 CRUD 중심
- 이후 slug + short_code 주소 체계 도입
- RPC 중심 저장 전략으로 데이터 일괄 교체 구조 형성

### account-book

- legacy key에서 현재 store key로 migration 로직 존재
- 단순 엔트리 리스트 구조에서 사용자/워크스페이스/공유 링크 개념으로 진화
- 최근에는 자연어 입력 중심 UX 개선이 진행 중

### daily

- localStorage 기반이지만 storage 추상화가 잘 되어 있어, 기능이 커지면서 구조도 함께 다듬어진 사례로 보임

### schedule

- service layer가 비교적 정돈되어 있어, 다른 메뉴보다 업무용 툴로 먼저 구조화되었을 가능성이 높음
- 별도 login/middleware가 붙어 있어 내부 운영 도구로 성격이 분화됨

## 5. 현재 구조가 말해주는 것

황총무는 "설계부터 완성된 제품"이라기보다,  
실사용 문제를 하나씩 해결하면서 **기능이 먼저 성장했고 구조가 뒤따라 정리되고 있는 제품**이다.

이건 약점만은 아니다.

장점:

- 실제 문제 중심으로 발전했다
- 과설계보다 실용성이 앞섰다

단점:

- 메뉴별 구현 품질 편차가 남는다
- 중복 패턴과 경계 모호성이 생긴다

---

## 6. 실무적으로 이 히스토리가 중요한 이유

이 문서는 단순 회고가 아니라 의사결정 기준을 제공한다.

예:

- 왜 지금 service layer 통일이 필요한가?
- 왜 localStorage 메뉴부터 백업 전략이 필요한가?
- 왜 slug routing이 있는 메뉴와 없는 메뉴가 갈리는가?

즉, 현재 상태를 고치기 전에 "왜 이렇게 됐는지"를 아는 것이 리팩토링 순서에 직접 영향을 준다.
