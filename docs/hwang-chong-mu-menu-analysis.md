# 황총무 프로젝트 메뉴별 상세 분석

작성 기준일: 2026-03-18  
분석 범위: 홈 화면에 노출되는 주요 메뉴와 그에 연결된 상세 라우트, 저장 방식, 공통 아키텍처, 보조 관리 화면까지 포함

## 1. 프로젝트 한눈에 보기

### 1-1. 프로젝트 정체성

황총무는 "복잡한 건 제가 할게요"라는 메시지에 맞게 일상과 업무의 번거로운 작업을 작은 도구 단위로 풀어낸 멀티 툴 프로젝트다.  
하나의 제품 안에 여러 개의 마이크로 서비스가 공존하는 구조이며, 홈 화면 카드가 각각의 독립 기능으로 연결된다.

핵심 특징은 아래와 같다.

- 여러 메뉴가 공존하지만 공통 헤더, 공통 레이아웃, 공통 버튼/모달/가이드 컴포넌트를 공유한다.
- 데이터 저장 전략이 메뉴마다 다르다.
  - Supabase 기반: 약속 잡기, N빵 계산기, 습관 관리, 체중 관리, 게임방, 업무 캘린더
  - localStorage 기반: 가계부, 일일 기록
- 각 메뉴는 "생성 페이지 -> 상세 페이지" 2단 분리 구조가 많다.
- URL은 사람이 읽기 쉬운 slug + short code 패턴을 일부 메뉴에서 채택하고 있다.
- 일정/정산처럼 메뉴 간 연결도 존재한다.
  - 대표 예: 약속 잡기에서 참석자 기준으로 N빵 계산기 방 생성

### 1-2. 기술 스택

- 프레임워크: Next.js App Router
- UI: React 19, styled-components, 일부 MUI 아이콘, 사내 공용 패키지 `@hwangchongmu/ui`
- 백엔드/DB: Supabase
- 날짜 처리: `date-fns`
- 전역 공통 처리:
  - `app/layout.tsx`에서 `StyledComponentsRegistry`, `ModalProvider`, `GlobalHeader` 주입
  - `middleware.ts`에서 `/schedule` 계열 보호

### 1-3. 프로젝트 구조적 성격

이 프로젝트는 엄격한 단일 도메인 앱이 아니라 "생활/업무 도구 모음집" 성격이 강하다.  
그래서 메뉴별로 구현 밀도와 패턴이 조금씩 다르며, 기능마다 최적의 저장 방식을 따로 채택한 흔적이 보인다.

이 구조의 장점:

- 실사용 문제를 빠르게 제품화하기 좋다.
- 기능별 실험이 자유롭다.
- 작은 기능을 독립적으로 개선하기 쉽다.

이 구조의 한계:

- 메뉴별 UI 톤과 상태 관리 방식이 완전히 통일되지는 않았다.
- Supabase와 localStorage가 혼합되어 있어 운영/백업/동기화 정책이 메뉴마다 달라진다.
- 일부 화면은 최근 리팩토링된 흔적이 있고, 일부는 초창기 패턴을 유지하고 있어 스타일 편차가 존재한다.

## 2. 홈 메뉴 맵

홈 화면 기준 주요 메뉴는 다음과 같다.

| 메뉴 | 진입 경로 | 상세/하위 경로 | 저장 방식 | 성격 |
| --- | --- | --- | --- | --- |
| 약속 잡기 | `/meeting` | `/meeting/room/[id]` | Supabase | 다인원 일정 조율 |
| N빵 계산기 | `/calc` | `/calc/[id]` | Supabase + RPC | 공통 비용 정산 |
| 가계부 | `/account-book` | `/account-book?workspaceId=...`, `/account-book/annual` | localStorage | 개인/공용 가계부 |
| 습관 관리 | `/habit` | `/habit/[id]` | Supabase | 월간 습관 추적 |
| 일일 기록 | `/daily` | `/daily/[id]` | localStorage | 체크리스트 + 일기 |
| 체중 관리 | `/diet` | `/diet/[id]` | Supabase | 식단/몸무게 로그 |
| 게임방 | `/game` | `/game/[id]`, `/game/quick/[id]` | Supabase + 일부 로컬 | 실시간/즉시 게임 |
| 업무 캘린더 | `/schedule` | `/schedule/create`, `/schedule/[id]`, `/schedule/[id]/kanban` | Supabase | 프로젝트 일정 관리 |
| 포트폴리오 | `/portfolio` | `/portfolio/experience/[id]`, `/ui-kit` | 로컬 데이터 | 소개/프로젝트 문서 |

보조 메뉴:

- `/login`: 업무 캘린더 접근용 관리자 비밀번호 화면
- `/ui-kit`: 공용 UI 문서 페이지

## 3. 공통 아키텍처 패턴

### 3-1. UI 공통 레이어

- `components/common`에 버튼, 모달, 공유, 가이드, 글로벌 헤더 등이 모여 있다.
- `components/styled/layout.styled.tsx`의 공통 래퍼를 여러 페이지가 재사용한다.
- `PageIntro`, `FooterGuide`, `CreateButton`은 생성형 메뉴들의 공통 문법 역할을 한다.

즉, "도구별 화면은 다르지만 진입 화면의 감성은 통일"하는 전략이다.

### 3-2. 데이터 접근 방식

메뉴별로 저장소가 두 갈래다.

#### Supabase형

- 생성/조회/업데이트/삭제 흐름이 페이지 + 훅 + 서비스로 분리된다.
- 예:
  - `hooks/useCreateRoom.ts`
  - `hooks/useRoom.ts`
  - `hooks/useCalcPersistence.ts`
  - `services/schedule.ts`

#### localStorage형

- 브라우저 전용 저장소를 직접 추상화한 storage 파일이 존재한다.
- 예:
  - `app/daily/storage.ts`
  - `app/account-book/storage.ts`

이 두 계층은 성격이 다르다.

- Supabase형은 멀티 유저/공유/실시간성에 강하다.
- localStorage형은 속도와 구현 단순성은 좋지만 디바이스 간 동기화는 없다.

### 3-3. 라우팅 전략

일부 메뉴는 UUID를 사람이 읽기 쉬운 주소로 리다이렉트한다.

- meeting: `slug-short_code`
- calc: `slug-short_code`

이 전략의 장점:

- 공유 링크가 덜 무섭다.
- 같은 방 링크를 브랜딩된 주소로 보이게 할 수 있다.

## 4. 메뉴별 상세 분석

## 4-1. 약속 잡기

### 목적

여러 사람이 가능한 날짜를 모아 최종 약속일을 정하는 협업형 일정 조율 도구다.  
핵심 아이디어는 "가능한 날짜를 고르는 방식"이 아니라 "안 되는 날짜를 제거하는 방식"이다.

### 주요 라우트

- 생성: `/meeting`
- 상세: `/meeting/room/[id]`

### 진입 구조

`/meeting`에서는 `RoomForm`과 `useCreateRoom`을 사용해 방을 만든다.

생성 시 입력 데이터:

- 방 이름
- 시작일
- 종료일 또는 자동 기간
- 주말 포함 여부

자동 기간 로직이 들어가 있어, 사용자가 종료일을 직접 넣지 않아도 일정 범위를 계산해준다.

### 상세 동작 구조

상세 페이지는 `useRoom` 훅이 사실상 핵심 엔진이다.

역할:

- 방 정보 조회
- 참가자 목록 조회
- 확정 날짜 여부 확인
- 투표 단계와 확정 단계 전환
- 불가능 날짜 저장
- 참가자 수정/삭제
- 최종 날짜 확정

UI는 `RoomHeader`, `CalendarGrid`, `ParticipantList`, `VoteSubmitButtons`, `FloatingFinishButton` 등으로 분리되어 있다.

### 데이터 모델 추정

Supabase 테이블/필드 관점에서 보면 아래 흐름이다.

- `rooms`
  - 이름
  - 시작/종료 날짜
  - 주말 포함 여부
  - slug, short_code
  - 확정 날짜
  - calc_room_id
- `participants`
  - room_id
  - name
  - unavailable_dates
  - is_absent

### 사용자 흐름

1. 방 생성
2. 참가자가 이름 입력
3. 불가능 날짜 선택
4. 참가자 목록 누적
5. 운영자가 후보일 비교
6. 최종 날짜 확정
7. 필요 시 N빵 계산기로 이동

### 이 메뉴의 강점

- "안 되는 날만 체크" UX가 매우 직관적이다.
- 이름 기반으로 쉽게 참여할 수 있다.
- 확정 이후 정산 방 생성까지 연결되어 실제 모임 흐름에 맞다.

### 기술적 특징

- 3초 polling 방식으로 참가자 데이터 갱신
- slug/short_code 리다이렉트 적용
- 최종 확정 후 `calc_room_id`를 통해 계산기와 연결
- 공통 모달 대신 훅 내부에서 자체 모달 상태도 함께 운영

### 아쉬운 점 / 개선 포인트

- polling 기반이라 사용량이 커지면 비효율적일 수 있다.
- 이름을 키로 일부 삭제/수정 로직이 동작해 동명이인 상황이 약할 수 있다.
- 참가자 권한/인증이 약해 공유 링크 환경에서 수정 통제가 제한적이다.

## 4-2. N빵 계산기

### 목적

공통 지출을 최소 이체 횟수로 정산해주는 계산 도구다.

### 주요 라우트

- 생성: `/calc`
- 상세: `/calc/[id]`

### 진입 구조

생성 페이지는 아주 간단하다.

- 모임 이름 입력
- `useCalcPersistence().createRoom()` 호출
- Supabase `calc_rooms` 생성
- slug + short_code 주소로 이동

### 상세 동작 구조

상세 페이지는 아래 세 축으로 동작한다.

1. `useCalcPersistence.fetchRoomData`
   - 멤버 목록
   - 지출 목록
   - 방 메타데이터 조회
2. `useCalculator`
   - 정산 계산 엔진
3. `CalcMainContent`
   - 멤버 관리
   - 지출 입력
   - 지출 리스트
   - 정산 리포트
   - 최종 송금 리스트
   - 공유 버튼

### 계산 로직 핵심

`useCalculator`는 다음을 계산한다.

- 공통 지출 합계
- 1인당 분담금
- 나머지 금액 처리
- 누가 누구에게 얼마 보내야 하는지

개인 지출과 공통 지출을 구분해 공통 지출만 분배 대상으로 삼는 구조가 좋다.

### 저장 방식

핵심은 Supabase RPC `calc_replace_room_data`다.

즉, 세부 멤버/지출 row를 프론트에서 하나씩 CRUD하기보다 "현재 방 상태 전체"를 서버 함수로 교체하는 구조에 가깝다.

장점:

- 저장 시 데이터 일관성을 맞추기 쉽다.
- 방 전체 상태를 한 번에 갱신할 수 있다.

주의점:

- 동시 편집 충돌 처리 전략은 문서상 분명하지 않다.

### 약속 잡기와의 연결

meeting에서 확정된 참석자 명단으로 calc 방을 생성하거나 기존 calc 방을 갱신하는 연결점이 있다.  
즉, 이 메뉴는 독립 도구이면서도 약속 잡기의 후속 단계 역할도 한다.

### 개선 포인트

- 정산 히스토리 버전 관리가 있으면 더 강력해질 수 있다.
- 멤버별 송금 완료 체크 같은 후속 관리 기능이 붙기 좋다.
- 실시간 공동편집에 대한 락/동기화 정책은 아직 약한 편이다.

## 4-3. 가계부

### 목적

개인 가계부와 공용 가계부를 함께 다루는 재무 기록 도구다.  
현재 구조상 단순 가계부를 넘어 "워크스페이스형 가계부 시스템"에 가깝다.

### 주요 라우트

- 허브: `/account-book`
- 연간 상세: `/account-book/annual`

### 저장 방식

localStorage 기반 자체 스토어를 사용한다.

스토어 구성:

- `users`
- `workspaces`
- `entries`
- `shareLinks`

즉, 브라우저 내부에서 다음을 모두 시뮬레이션한다.

- 사용자 계정
- 개인/공용 가계부방
- 수입/지출 엔트리
- 엔트리 공유 링크

### 핵심 구조

`/account-book`는 허브와 상세 뷰를 분기한다.

- 허브: `WorkspaceHub`
- 방 상세: `WorkspaceLedgerView`
- 설정: `WorkspaceSettingsModal`
- 잠금: `AccountBookLockGate`

이 구조는 단순한 단일 리스트보다 훨씬 고차원적이다.

### 현재 UX 특징

최근 리팩토링을 통해 자연어 입력 중심으로 크게 단순화된 흔적이 보인다.

예:

- "3월 17일 네이버쇼핑 마라샹궈 구매 16만원" 같은 문장을 파싱
- 날짜/가맹점/항목/결제수단/금액으로 해석
- 월별 리스트/캘린더/보드로 조회

즉, 내부 데이터 모델은 복잡하지만 입력 경험은 최대한 단순화하려는 방향이다.

### 조회 레이어

`resolveWorkspaceEntries`가 중요한 역할을 한다.

- 직접 작성한 엔트리
- 공유 링크로 들어온 엔트리
- 공용방에서 개인방으로 미러링된 엔트리

이 세 가지를 합성해서 최종 표시용 엔트리 리스트를 만든다.

### 장점

- 개인/공용 회계를 동시에 다루는 구조적 확장성이 좋다.
- 비밀번호, 공유, 연간 요약, 월별 보드 등 기능 폭이 넓다.
- 기존 레거시 localStorage 데이터 마이그레이션도 고려되어 있다.

### 한계

- 기능이 많아 학습 비용이 높은 편이다.
- localStorage 기반이라 기기 변경/브라우저 삭제 시 데이터 유실 가능성이 있다.
- 다중 사용자처럼 보이지만 실제 협업 동기화는 로컬 환경 한정이다.

### 개선 포인트

- 장기적으로는 Supabase 동기화 또는 export/import 지원이 매우 중요하다.
- 자연어 입력 정확도를 높일 여지가 크다.
- 워크스페이스/공유 구조를 실제 사용자 개념에 맞게 더 단순화할 수 있다.

## 4-4. 습관 관리

### 목적

하루 단위 체크를 월간 달력과 결합해 습관을 시각적으로 추적하는 서비스다.

### 주요 라우트

- 생성: `/habit`
- 상세: `/habit/[id]`

### 저장 방식

Supabase 기반.

주요 테이블 추정:

- `goals`
- `goal_items`
- `goal_logs`

### 구조

생성 페이지에서 목표 제목, 이모지, 테마 컬러를 받아 `goals`를 만든다.

상세 페이지에서는 `MonthlyTracker`가 핵심 컨테이너다.
그 안에서 다시 `useMonthlyTracker`가 다음을 관리한다.

- 현재 월
- 선택 날짜
- 주말 표시 여부
- 목표 항목 목록
- 월별 로그
- 선택 일자의 완료 상태
- hover 상태

### 화면 구성

- 좌측: 월간/주간 달력 + 할 일 목록
- 우측: 댓글 + 월간 랭킹

즉, 단순 체크리스트가 아니라 커뮤니티/경쟁 요소가 약하게 섞인 구조다.

### 좋은 점

- 달력 heatmap 성격이 강해서 성취감이 보인다.
- 주말 표시 토글과 월/주 전환이 실사용에 좋다.
- 아이템 단위 완수 로그라 확장 가능성이 높다.

### 아쉬운 점

- 습관 그룹/카테고리 개념은 아직 약하다.
- 데이터가 월 단위 시각화에 강하지만 장기 추세 화면은 상대적으로 약하다.
- 댓글, 랭킹, 체크 기능이 한 화면에 모여 있어 처음엔 정보량이 많을 수 있다.

## 4-5. 일일 기록

### 목적

한 줄 일기와 체크리스트를 한 곳에 담는 "개인 기록장" 서비스다.

### 주요 라우트

- 생성: `/daily`
- 상세: `/daily/[id]`

### 저장 방식

localStorage 기반이며, 구현 밀도가 높다.

저장 구조:

- 기록장 메타: `daily-notebooks`
- 기록장별 엔트리: `daily-notebook-entries:${id}`

### 구조적 특징

이 메뉴는 localStorage형 서비스 중 가장 체계적이다.

- 기록장 메타와 월별 엔트리를 분리 저장
- 월별 체크리스트 커스터마이징 지원
- 기록장 잠금용 accessCode 지원
- sessionStorage 기반 잠금 해제 상태 저장
- 테마 컬러 변경 지원

### 상세 페이지 핵심 기능

- 월 이동
- 해당 월 날짜 자동 생성
- 날짜별 일기 작성
- 체크 항목 O/X 기록
- 평균 점수 계산
- 월간 trend graph 시각화
- 기록장 비밀번호 설정/해제

### 이 메뉴의 강점

- 서버 없이도 개인 기록장 제품 경험을 꽤 완성도 높게 제공한다.
- 월별 체크리스트가 달라질 수 있어 현실적이다.
- "오늘" 중심이 아니라 월 단위 회고가 가능하다.

### 한계

- localStorage 기반이라 백업/동기화 없음
- 모바일 간 이전이 불편
- 협업이 아니라 철저한 개인 기록 중심

### 개선 포인트

- export/import
- 캘린더 요약 보기
- 주간/월간 회고 템플릿
- 클라우드 동기화

## 4-6. 체중 관리

### 목적

몸무게와 식단을 함께 기록하며 주간/월간 추세를 보는 건강 기록 도구다.

### 주요 라우트

- 생성: `/diet`
- 상세: `/diet/[id]`

### 저장 방식

Supabase 기반.

주요 테이블 추정:

- `diet_goals`
- `diet_logs`

### 핵심 구조

상세 페이지는 `DietMainContent`가 거의 전부를 담당한다.

주요 상태:

- 현재 날짜
- 주간/월간 보기 모드
- 오늘 로그
- 어제 로그
- 차트용 기간 로그
- dirty 상태

### 기능 포인트

- 아침/점심/저녁 식단 분리 입력
- 시간대별 몸무게 입력
- 메모 작성
- 어제 저녁 대비 / 오늘 아침 대비 변화량 계산
- 주간/월간 차트 조회
- 저장 전 이탈 경고

### 강점

- 식단 + 체중을 같은 도메인 안에서 본다.
- 단순 몸무게 숫자 기록보다 맥락이 풍부하다.
- 건강 변화 분석 카드가 붙기 좋은 구조다.

### 한계

- 운동/수면/컨디션 같은 보조 지표는 아직 약하다.
- 식단 태그가 문자열 기반이라 구조화된 분석에는 한계가 있다.

## 4-7. 게임방

### 목적

친구들과 간단한 복불복 게임이나 즉석 게임을 즐기는 놀이 도구다.

### 주요 라우트

- 로비: `/game`
- 방 게임: `/game/[id]`
- 빠른 시작: `/game/quick/[id]`

### 구조적 특징

게임방은 두 모드를 동시에 품고 있다.

1. 빠른 시작
   - 로컬 상태 기반
   - 참가자 추가 후 즉시 게임 시작
2. 방 만들기
   - Supabase 기반 실시간 참가
   - 친구 초대형 게임

### 로비 설계

`viewMode`에 따라

- SELECT
- QUICK_LIST
- CREATE

세 상태를 전환한다.

즉, 단일 페이지 안에서 게임 포털 역할을 한다.

### 상세 방 구조

`useGameRoom` 훅이 핵심이다.

- 방 정보 조회
- 참가자 조회
- 내 참가 상태 판별
- 호스트 여부 판별
- 게임 종류 선택
- 게임 시작/종료
- 실시간 구독

실시간은 Supabase channel 기반 postgres_changes를 사용한다.

### 지원 게임

현재 코드상 주요 게임:

- 텔레파시
- 광클 대전
- 돌림판
- 사다리 타기

빠른 시작은 주로 로컬형 게임에 집중되고, 방형 게임은 실시간 룸 로직을 탄다.

### 좋은 점

- 즉시성 모드와 실시간 모드를 모두 제공한다.
- 실시간 구독 구조를 이미 갖고 있다.
- 참여자 관리 UX가 직관적이다.

### 주의점

- 비밀번호와 참가자 권한이 강력한 인증 체계는 아니다.
- 게임마다 상태 구조가 다를 가능성이 높아, 공통 엔진이 더 필요해질 수 있다.
- 로컬 모드와 서버 모드의 코드 일관성이 장기적으로 과제가 될 수 있다.

## 4-8. 업무 캘린더

### 목적

서비스/프로젝트별 일정을 보드 단위로 관리하는 업무용 일정 시스템이다.

### 주요 라우트

- 목록: `/schedule`
- 생성: `/schedule/create`
- 상세 캘린더: `/schedule/[id]`
- 칸반: `/schedule/[id]/kanban`
- 로그인: `/login`

### 접근 제어

이 메뉴만 `middleware.ts`로 별도 보호된다.

- `/schedule` 하위 진입 시 `auth_token` 쿠키 검사
- 없으면 `/login`으로 리다이렉트

즉, 프로젝트 전체에서 유일하게 명시적 보호 레이어가 있는 메뉴다.

### 데이터 모델

`services/schedule.ts` 기준으로 추정하면 다음 3층 구조다.

- `schedule_boards`
- `schedule_services`
- `schedule_tasks`

보드 아래 서비스가 있고, 서비스 아래 태스크가 있는 구조다.

### 목록 페이지

- 보드 목록 조회
- 보드 수정/삭제
- 보드 생성 진입

`ServiceCard`를 통해 보드 카탈로그 느낌으로 동작한다.

### 상세 캘린더 페이지

화면은 좌우 분할 구조다.

- 좌측: 월간 캘린더 + 드래그성 이동
- 우측: 서비스별 태스크 패널

핵심 기능:

- 보드와 서비스/태스크 로딩
- 주말 표시 여부 토글
- 숨김 서비스 관리
- 태스크 날짜 이동
- 포커스 복귀 시 재조회

### 칸반 페이지

같은 보드를 To-Do / In Progress / Done으로 재해석한다.

분류 기준:

- 완료 여부
- 태스크 존재 여부

즉, 같은 데이터셋을 "캘린더 뷰"와 "칸반 뷰"로 각각 소비한다.

### 장점

- 일정 보기와 프로젝트 진행 보기 두 관점이 분리되어 있다.
- 서비스 계층과 태스크 계층이 있어 실제 업무 구조와 잘 맞는다.
- 업무 메뉴만 별도 접근 제어가 있는 점이 명확하다.

### 한계

- 인증이 쿠키 플래그 수준이라 보안적으로는 단순하다.
- 서비스/태스크 편집 흐름이 많아질수록 페이지 복잡도가 상승할 수 있다.
- 실시간 협업보다는 "복귀 시 새로고침" 성격이 강하다.

## 4-9. 포트폴리오

### 목적

개발자 소개, 이력, 실무 경험, 토이 프로젝트 문서를 한 곳에 정리한 개인 포트폴리오다.

### 주요 라우트

- 메인: `/portfolio`
- 경험 상세: `/portfolio/experience/[id]`
- 보조 문서: `/ui-kit`

### 데이터 성격

이 메뉴는 Supabase보다는 로컬 정적 데이터 중심이다.

- `data/experiences.tsx`
- `data/constants.ts`
- 프로젝트 이미지 자산

### 구성

- `PortfolioInfo`: 소개/외부 링크/프로필
- `ResumeSection`: 이력 요약
- `ProjectSection`: 토이 프로젝트와 기술 스택
- `ProjectList`: 실제 프로젝트 카드
- `/ui-kit`: 공용 UI 문서 페이지

### 특징

- 포트폴리오 안에 황총무의 토이 프로젝트를 다시 소개하는 메타 구조다.
- UI Kit을 별도 문서 사이트처럼 보여주는 점이 좋다.
- 실무 경력과 토이 프로젝트가 한 문맥으로 이어진다.

### 한계

- 데이터가 코드 상수에 강하게 묶여 있어 CMS형 관리와는 거리가 있다.
- 콘텐츠가 늘어나면 에디터형 관리 도구가 필요해질 수 있다.

## 5. 보조 메뉴 분석

## 5-1. 로그인 페이지

`/login`은 사실상 업무 캘린더 전용 게이트다.

- 환경변수 `NEXT_PUBLIC_ADMIN_PASSWORD` 비교
- 통과 시 `auth_token=true` 쿠키 저장
- `/schedule`로 이동

보안 강도는 낮지만, 개인 프로젝트 내부 운영 도구용 보호막으로는 빠르게 작동한다.

## 5-2. UI Kit 페이지

`/ui-kit`은 공용 컴포넌트 문서 사이트 역할을 한다.

- 좌측 내비게이션
- 가운데 문서 본문
- 우측 TOC
- 코드 복사 기능

즉, 황총무 프로젝트 안에서 디자인 시스템을 분리해 정리하려는 의도가 반영된 보조 메뉴다.

## 6. 메뉴 간 연결 관계

### 직접 연결

- 약속 잡기 -> N빵 계산기
  - 확정 참석자 기준으로 정산 방 생성
- 포트폴리오 -> UI Kit
  - 문서형 연결

### 간접 연결

- 홈 메뉴는 각각 독립 도구지만 시각 언어는 공통 컴포넌트로 연결
- 습관/일일 기록/체중 관리는 모두 "개인 기록" 카테고리로 묶일 수 있음
- 업무 캘린더는 다른 메뉴보다 운영 도구 성격이 강함

## 7. 메뉴별 성숙도 평가

### 성숙도가 높은 축

- 약속 잡기
  - 생성 -> 투표 -> 확정 -> 정산 연결까지 흐름이 명확
- N빵 계산기
  - 도메인 목적이 선명하고 계산 엔진이 명확
- 일일 기록
  - localStorage 기반이지만 기능 완성도가 높음
- 업무 캘린더
  - 데이터 계층과 화면 분리가 가장 업무툴답게 잡혀 있음

### 확장 중인 축

- 가계부
  - 구조는 크고 강하지만 UX 단순화가 진행 중
- 게임방
  - 재미 요소는 좋지만 공통 게임 엔진 성숙도는 더 올라갈 수 있음
- 습관 관리
  - 사용성 좋지만 장기 통계와 커뮤니티 확장은 여지 많음
- 체중 관리
  - 건강 로그로서 좋지만 분석 축 확장 여지가 큼

## 8. 프로젝트 전반 강점

- 실제 생활 문제를 해결하는 툴셋이라 기획 의도가 명확하다.
- 메뉴별 도메인 경계가 비교적 뚜렷하다.
- 공통 UI 컴포넌트를 재사용하려는 방향성이 분명하다.
- slug/short_code, RPC, localStorage 추상화 등 실용적인 기술 선택이 좋다.
- "토이 프로젝트"지만 실제 사용을 염두에 둔 흐름 설계가 많다.

## 9. 프로젝트 전반 리스크

- 저장 전략 이원화
  - 같은 제품 안에서 Supabase와 localStorage가 혼재
- 보안 정책 편차
  - schedule만 미들웨어 보호, 나머지는 상대적으로 자유로움
- 코드 스타일 편차
  - 초기 페이지와 최근 리팩토링 페이지의 패턴 차이 존재
- 타입 엄격성 편차
  - 일부 `any`와 문자열 기반 상태 사용 흔적
- 실시간 처리 정책 편차
  - 어떤 메뉴는 polling, 어떤 메뉴는 realtime, 어떤 메뉴는 로컬 only

## 10. 추천 개선 우선순위

### 1순위: 저장/동기화 전략 정리

- 개인 기록형(localStorage) 메뉴에 export/import 추가
- 가계부와 일일 기록에 백업 정책 도입

### 2순위: 메뉴별 UX 통일

- 생성 페이지 공통 패턴 통일
- 상세 페이지 헤더/도움말/공유 UX 정리

### 3순위: 권한과 접근 제어 정리

- schedule 외 관리성 메뉴의 보호 정책 검토
- 링크 공유형 메뉴의 수정 권한 설계 강화

### 4순위: 공통 상태/데이터 계층 정리

- Supabase 메뉴의 service/hook 패턴 통일
- polling/realtime 기준 정립

### 5순위: 포트폴리오와 제품 문서 연결 강화

- 각 메뉴에 "왜 만들었는지 / 어떤 기술 문제를 풀었는지" 연결
- UI Kit, Portfolio, Product Docs를 하나의 개발자 서사로 묶기

## 11. 결론

황총무는 단순한 토이 묶음이 아니라, "생활형 문제 해결 도구 모음집"이라는 제품 콘셉트가 분명한 프로젝트다.  
특히 약속 잡기, N빵 계산기, 업무 캘린더, 일일 기록은 실제 사용자 흐름과 저장 구조가 비교적 잘 정리되어 있고,  
가계부와 게임방은 현재 확장성과 UX를 함께 실험하는 단계로 보인다.

가장 큰 구조적 포인트는 다음 세 가지다.

1. 메뉴별 독립성이 높다.
2. 실사용 문제 해결력이 강하다.
3. 이제는 "기능 추가"보다 "일관성/동기화/권한 체계 정리" 단계로 넘어갈 시점이다.

원한다면 다음 단계로는 이 문서를 바탕으로 아래 문서도 추가 작성할 수 있다.

- 메뉴별 DB 테이블 관계도
- 메뉴별 사용자 플로우 차트
- 리팩토링 우선순위 문서
- Supabase/localStorage 전환 전략 문서
