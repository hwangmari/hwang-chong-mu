# N빵 계산기 상세 분석

## 1. 메뉴 목적

N빵 계산기는 공통 비용을 최소 이체 횟수로 정산하는 도구다.  
황총무 안에서는 독립 기능이면서도, 약속 잡기 이후 단계와도 강하게 연결되는 메뉴다.

## 2. 라우트 구조

- 생성 페이지: `/calc`
- 상세 페이지: `/calc/[id]`

## 3. 핵심 파일 구조

```text
app/calc/page.tsx
hooks/useCalcPersistence.ts
app/calc/[id]/page.tsx
hooks/useCalculator.ts
app/calc/components/*
```

핵심 레이어는 아래처럼 나뉜다.

- 생성/저장 담당: `useCalcPersistence`
- 계산 엔진: `useCalculator`
- 화면 조합: `CalcMainContent`

## 4. 생성 페이지 분석

생성 페이지는 매우 단순하다.

- 모임 이름 입력
- 정산 방 생성
- 생성 즉시 상세 페이지 이동

이 단순함은 장점이다.  
정산 서비스는 도메인 자체가 명확해서, 생성 단계에서 많은 옵션이 필요하지 않다.

## 5. 상세 페이지 분석

상세 페이지는 다음 흐름으로 구성된다.

1. URL로 방 조회
2. 멤버 목록 로딩
3. 지출 목록 로딩
4. 계산 엔진으로 정산 결과 산출
5. 화면에 섹션별 출력

`CalcMainContent` 내부 섹션:

- `MemberManager`
- `ExpenseInput`
- `ExpenseList`
- `SettlementReport`
- `SettlementList`
- `ShareButton`

이 구조는 "입력 -> 목록 -> 결과 -> 공유" 순서가 명확하다.

## 6. 계산 로직 분석

`useCalculator`는 이 메뉴의 핵심 가치다.

### 입력

- 멤버 배열
- 지출 배열
- 정산 단위

### 처리

- 공통 지출만 필터링
- 1인당 분담금 계산
- 나머지 금액 처리
- 각 멤버의 실질 잔액 계산
- 받는 사람 / 보내는 사람 분리
- 최소 이체 횟수로 settlement 리스트 생성

### 출력

- 총 공통 지출
- 1인당 부담액
- 이체 리스트
- 나머지 금액과 수령자

이 로직은 정산기에서 중요한 "누가 누구에게 얼마"를 직접 만든다.

## 7. 저장 방식 분석

`useCalcPersistence`에서 중요한 부분은 RPC 기반 저장이다.

사용 API:

- `calc_rooms` 생성
- `calc_members` 조회
- `calc_expenses` 조회
- `calc_replace_room_data` RPC로 전체 상태 교체

즉, 이 메뉴는 row 단위 CRUD보다 "방 전체 상태 스냅샷 갱신"에 가깝다.

### 장점

- 프론트 상태와 DB 상태를 맞추기 쉽다
- 멤버와 지출을 일괄 갱신하기 좋다

### 한계

- 동시 수정 충돌 처리 정책이 필요해질 수 있다
- 변경 이력/되돌리기 기능이 없다

## 8. 사용자 시나리오

1. 방 생성
2. 멤버 추가
3. 지출 입력
4. 공통/개인 지출 구분
5. 정산 결과 확인
6. 결과 복사 또는 공유

약속 잡기에서 넘어온 경우:

1. 약속 확정
2. 참석자 기준 정산 방 자동 생성
3. 비용만 추가 입력

## 9. 기술적 특징

- slug + short_code 링크 구조
- 자동 저장 성격이 강함
- 정산 계산이 hook으로 분리되어 테스트 가능성이 좋음
- UI와 계산 로직이 분리되어 있음

## 10. 강점

- 도메인 경계가 매우 선명하다
- 사용자 기대 결과가 명확하다
- 계산 로직이 훅으로 분리되어 구조가 깔끔하다
- 약속 잡기와 연결되며 제품 흐름이 자연스럽다

## 11. 한계

- 송금 완료 체크 같은 후속 액션 레이어가 부족하다
- 공동 편집 충돌에 대한 설계는 약하다
- 지출 카테고리 분석이나 회고 기능은 없다

## 12. 개선 제안

- 송금 완료 상태 관리
- 지출 카테고리별 집계
- 변경 이력 / 되돌리기
- 멤버별 링크 공유와 개인 확인 UI
- 정산 리포트 export 기능
