# 접근 제어 및 보안 메모

상태: Working Draft  
작성일: 2026-03-18

---

## 1. 목적

이 문서는 황총무 프로젝트의 현재 접근 제어 수준과 보안상 주의할 포인트를 기록한다.

주의:

현재 프로젝트는 본격적인 엔터프라이즈 보안 모델을 갖춘 상태는 아니며, 일부 메뉴는 개인용/실험용 수준의 보호만 제공한다.

---

## 2. 현재 접근 제어 수준

### 2-1. 업무 캘린더

보호 방식:

- `middleware.ts`
- `/login`
- `auth_token` 쿠키

특징:

- 프로젝트 내에서 가장 명시적인 보호 레이어 보유
- 하지만 쿠키 값 자체는 단순 플래그 수준

### 2-2. 가계부

보호 방식:

- 워크스페이스 비밀번호
- 설정 비밀번호
- session storage 기반 접근 플래그

특징:

- UX 차원의 잠금은 있음
- 서버 검증은 없음

### 2-3. 게임방

보호 방식:

- 참가자별 비밀번호
- localStorage 기반 재입장 보조 정보

특징:

- 가벼운 참여 통제 수준
- 강한 인증/권한 모델은 아님

### 2-4. 기타 메뉴

- 습관 / 체중 / 일일 기록 / 약속 잡기 / 계산기 대부분은 링크 또는 ID 중심 접근
- 엄격한 인증 계층은 없음

---

## 3. 현재 보안 리스크

### 리스크 1. 환경 변수 노출 경계

- `NEXT_PUBLIC_*`는 클라이언트에 노출됨
- 관리자 비밀번호를 public env에 두는 방식은 최소 보호 수준에 그침

### 리스크 2. localStorage / sessionStorage 의존 보호

- 브라우저 단 보호이므로 본질적 보안이 아님

### 리스크 3. 링크 기반 접근

- URL만 알면 접근 가능한 메뉴가 많음

---

## 4. 운영상 해석

현재 보호 수준은 아래처럼 해석하는 것이 현실적이다.

- 개인/가정/실험용 수준: 충분
- 민감한 실서비스 운영 수준: 부족

---

## 5. 권장 개선

### 단기

- 보안 민감 메뉴 목록 정의
- "이 기능은 브라우저 로컬 저장입니다" 안내 강화
- public env에 있는 민감값 재검토

### 중기

- schedule 관리자 인증 구조 개선
- 메뉴별 접근 정책 문서화

### 장기

- 사용자 계정 체계
- 서버 검증 기반 보호
- role 기반 접근 제어
