# 리팩토링 우선순위

상태: Active  
작성일: 2026-03-18  
목적: 황총무 프로젝트의 기술부채와 개선 대상을 우선순위 기준으로 정리

---

## 1. 우선순위 산정 기준

우선순위는 아래 기준으로 판단한다.

- 사용자 영향도
- 데이터 유실 가능성
- 구조적 파급력
- 공통화 가치
- 리팩토링 난이도

---

## 2. P0 - 가장 먼저 다뤄야 하는 영역

### P0-1. localStorage 기반 메뉴 백업 전략

대상:

- 가계부
- 일일 기록

이유:

- 데이터 유실 리스크가 직접적
- 사용자 신뢰에 영향

권장 액션:

- export/import
- JSON 다운로드
- 복구 가이드

### P0-2. 문서/구조 표준화

이유:

- 메뉴가 많아 온보딩 비용이 커짐
- 구조 편차가 누적될수록 유지보수 비용 증가

권장 액션:

- 현재 문서 세트 유지
- 새 기능 추가 시 문서 동반 갱신 규칙 정착

---

## 3. P1 - 공통 레이어 정리

### P1-1. Supabase 접근 패턴 통일

현재 상태:

- 어떤 메뉴는 훅에서 직접 호출
- 어떤 메뉴는 service 레이어 사용
- 어떤 메뉴는 RPC 중심

권장 방향:

- 최소한 service/repository 경계 문서화
- 새 메뉴는 service 레이어 우선

### P1-2. 컨테이너 / 프레젠테이션 분리 강화

우선 대상:

- diet
- daily
- 일부 game 화면

이유:

- 컨테이너가 두꺼워질수록 테스트/수정 난도 상승

---

## 4. P2 - 사용자 경험과 운영 품질 향상

### P2-1. 가계부 자연어 입력 고도화

이유:

- 이 메뉴의 핵심 차별점
- 입력 정확도가 제품 체감 품질에 직접 연결

### P2-2. meeting 실시간화 검토

현재:

- polling 기반

개선:

- realtime 또는 보다 명시적인 새로고침 전략

### P2-3. schedule 권한 체계 강화

이유:

- 내부 도구 성격이 강함
- 현재 보호 수준은 단순 플래그형

---

## 5. P3 - 장기 개선

### P3-1. 사용자 계정 중심 통합 전략

질문:

- 황총무를 단일 사용자 제품으로 볼 것인가?
- 계정 기반 멀티 디바이스 앱으로 갈 것인가?

이 결정에 따라 아래가 모두 달라진다.

- 저장 전략
- 인증 방식
- 공유 정책
- 동기화 전략

### P3-2. 테스트 전략 수립

필요 항목:

- 핵심 계산 로직 테스트
- storage 마이그레이션 테스트
- slug/shortcode 파싱 테스트
- RPC 호출 경계 테스트

---

## 6. 메뉴별 우선 개선 후보

### meeting

- polling 구조 재검토
- 참가자 식별 강화

### calc

- 저장 충돌 정책
- 결과 히스토리

### account-book

- 백업/동기화
- 워크스페이스 모델 단순화

### habit

- 장기 통계
- 목표 구조 확장

### daily

- export/import
- 페이지 로직 분리

### diet

- 운동/수면/컨디션 지표
- 컨테이너 경량화

### game

- 공통 게임 엔진 인터페이스
- 방 모드와 로컬 모드 구조 정리

### schedule

- 권한 모델 강화
- service layer 확장

---

## 7. 권장 실행 순서

1. 문서 체계 정착
2. localStorage 메뉴 백업 기능
3. Supabase 접근 패턴 정리
4. 가장 두꺼운 컨테이너 분리
5. 보안/접근 제어 정비

---

## 8. 결론

황총무의 다음 단계는 기능 추가보다 **신뢰성, 일관성, 운영성 강화**에 가깝다.  
즉, 지금 필요한 리팩토링은 "예쁘게 다시 쓰기"가 아니라,  
`저장 전략 정리 -> 공통 구조 정리 -> 운영 안정성 강화` 순서로 접근해야 한다.
